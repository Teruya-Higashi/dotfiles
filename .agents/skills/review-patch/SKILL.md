# Review Patch

diff の表面的なレビューではなく、変更の背景と影響範囲を理解した上で指摘する。

**REQUIRED SUB-SKILL:** PR 番号・URLを扱う場合や、GitHub からの情報取得・レビュー投稿などで `gh` を使う場合は、`gh-ops` を読み、そのルールに従う。

## 入力

引数でレビュー対象を指定する。省略時は現在のブランチのデフォルトブランチ（`main` 等）からの差分をレビューする。

| 引数 | 例 | 対象 |
|------|-----|------|
| PR 番号 / URL | `5612`, `https://...pull/5612` | PR の差分 |
| ブランチ名 | `feature/foo` | `main...feature/foo` の差分 |
| コミット範囲 | `abc123..def456` | 指定範囲の差分 |
| なし | | `main...HEAD` の差分 |

デフォルトブランチが `main` 以外（`master` / `develop` 等）の場合はそれに読み替える。

## Step 0: 対象の取得

### 0a. PR 情報の取得（PR 指定時のみ）

```bash
PR_NUMBER=... # 引数から抽出
gh pr view $PR_NUMBER --json headRefName,baseRefName,title,body,url,additions,deletions,changedFiles
# → BRANCH=headRefName, BASE=baseRefName を以降で使用
```

### 0b. 差分の取得

| 対象 | 取得方法 |
|------|---------|
| PR | `gh pr diff $PR_NUMBER` でパッチ、`gh pr diff $PR_NUMBER --name-only` でファイル一覧 |
| ブランチ | `git diff $BASE...{branch} --stat` と `git diff $BASE...{branch}` |
| コミット範囲 | `git diff abc123..def456` |
| なし | `git diff $BASE...HEAD` |

PR の head ブランチがローカルに無く、影響範囲調査（Phase 1b）でコードを読む必要がある場合は、先に取得しておく:

```bash
git fetch origin $BRANCH
```

### 0c. 既存レビューの確認（PR 指定時のみ）

CI bot や他のレビュアーが残した既存レビューコメントと、その会話スレッドを取得する:

```bash
gh api repos/:owner/:repo/pulls/$PR_NUMBER/comments --jq '.[] | {path, line, body, user: .user.login, in_reply_to_id, id}'
```

取得したコメントと返信を分析し、以下のルールで重複を排除する:

- 既存の指摘に対して **PR 作者が返信・修正対応済み** のスレッドは、その論点について指摘しない
- 既存で指摘済みかつ **未返信・未解決** のスレッドも、同じ内容を重複指摘しない
- 既存レビューが見逃している観点や、既存の指摘では不十分な点のみコメントする
- 全ての指摘が既存レビューでカバー済みの場合は「既存レビューで網羅済み、追加指摘なし」とユーザーに報告し、GitHub への投稿はスキップする

**重複判定はセマンティック**: 表面的な文字列一致ではなく「論点（観点・根拠・修正方針）」で判定する。言い回しが違っても、指摘する問題の本質と修正案が一致していれば重複とみなす。長文コンテキストの保持力を活かし、複数スレッドを横断して論点マッピングを行うこと。

## 3フェーズレビュー

### Phase 1: コンテキスト収集（レビュー前に必ず実施）

レビューを始める前に、以下を**すべて**収集する。このフェーズを省略しない。

**実行方針**: 1a–1d の情報収集は独立しているため、可能な限り**単一メッセージ内の並列 tool call** で実行すること。PR description の取得（`gh pr view`）、影響範囲の Grep、関連ルールの Read、テストファイルの Glob はそれぞれ独立しており、逐次発行する理由はない。並列発行で Phase 1 全体のレイテンシを短縮する。

#### 1a. 変更の意図を把握

- PR description / linked issue を読む
- コミットメッセージの流れを確認する
- 「なぜこの変更が必要か」を言語化できる状態にする

#### 1b. 影響範囲の調査

変更されたシンボル（関数、型、インターフェース等）について:

- **呼び出し元**: この関数/メソッドを呼んでいる箇所
- **参照元**: この型/インターフェースを使っている箇所
- **依存先**: この変更が依存しているモジュール

**調査深度の目安**（1M context を前提）:

| モード | 深度 | 理由 |
|--------|------|------|
| インラインモード | **2 hop まで**（呼び出し元の呼び出し元も辿る） | メインコンテキストに関連コードを全て保持できるため、より広い射程で影響を検証できる |
| 並列エージェントモード | 1 hop（エージェント側で必要に応じて追加調査） | エージェントのコンテキストを圧迫しないため |

コードインデックス系の MCP（Serena 等）が使える場合は `find_referencing_symbols` を活用する。使えない場合は Grep で調査する。

#### 1c. 関連ルールの特定

変更内容に応じて、プロジェクトの規約から関連するものを特定し、**内容を読み込む**。全てを毎回読むのではなく、変更に関係するものだけを選択する。

規約の所在はプロジェクトによって異なる。以下を確認する:

- `.claude/rules/` 配下のルール
- `AGENTS.md` / `CLAUDE.md`（リポジトリルート、または対象ディレクトリ）
- 言語・領域別の関連 skill（あれば）
- `CONTRIBUTING.md` / `.editorconfig` / linter 設定

| 変更対象 | 参照する規約の例 |
|---------|-------------|
| アプリケーションコード（Go / TypeScript 等） | 言語別コーディング規約 |
| フロントエンド | UI / コンポーネント規約 |
| インフラ（Terraform 等） | インフラ規約 |
| テスト変更 | テスト・検証規約 |

#### 1d. テストの確認

- 変更に対応するテストが追加/更新されているか
- 既存テストが変更の影響で壊れていないか

#### Phase 1 の中間出力

Phase 1 完了時点で以下を出力し、そのまま Phase 2 へ継続する（立ち止まらない）。レビュー結果と合わせて提示することで、ユーザーが前提のズレを事後検証できるようにする。

```
### コンテキスト収集結果
- **変更の意図**: {1a で把握した内容}
- **影響範囲**: {1b の調査結果の要約}
- **関連ルール**: {1c で特定したルール一覧}
- **テスト**: {1d の確認結果}
```

### Phase 2: レビュー

#### 実行モードの選択

diff の規模に応じて実行モードを選択する。

| 条件 | モード | 理由 |
|------|--------|------|
| 変更ファイル 10 以下 かつ 追加+削除 500 行以下 | **インライン** | メインが全観点を1パスでレビュー。コンテキスト分断なし（1M context を前提に閾値を緩和） |
| 変更ファイル 10 超 または 追加+削除 500 行超 | **並列エージェント** | 注意力希釈を防ぐため専門エージェントに分割 |

**例外**: 上記で並列に倒れる規模でも、Phase 1b の影響範囲調査がほぼ1 hop で完結し関連コードを合わせて 1M context に余裕で収まる場合は、メインの判断でインラインに切り替えてよい（Phase 1 のコンテキストを Phase 2–3 まで保持できる利点があるため）。

#### インラインモード

エージェントを起動せず、メインエージェントが全観点（正確性・セキュリティ・設計+再利用・効率）を1パスで直接レビューする。指摘候補は並列エージェントモードと同じフォーマットでリストアップし、Phase 2.5 → Phase 3 へ進む。

**思考方式**: 1回の拡張思考（think harder / ultrathink）パスで4観点を横断的に検証する。観点ごとに Read をやり直さず、Phase 1 で読み込んだコンテキストを再利用する。後述の「エージェントへの共通指示」の「思考方針」セクションをメイン自身も適用すること。

#### 並列エージェントモード

Phase 1 で収集したコンテキストを入力として、**専門エージェントを Agent ツールで並列起動する**。

**エージェント起動パラメータ**:

```
subagent_type: general-purpose
model: opus          # 推論ヘビーなレビュー観点を正確に判定するため
```

- すべての専門エージェント（正確性・セキュリティ・設計+再利用・効率）で上記を**明示的に指定**する。Agent ツールは model 省略時に親モデルを継承するが、軽量モデルへ暗黙的にフォールバックするリスクを避けるため明示する
- 観点が限定的で推論負荷が軽い場合（テストコードのみ、CI/CD 設定のみ等）は `model: sonnet` を許容（コスト最適化）。ただしデフォルトは opus
- 並列起動は**単一メッセージ内に複数 Agent 呼び出しを並べる**こと。逐次起動しない

| 観点 | デフォルトモデル | Sonnet 許容の条件 |
|------|----------------|-----------------|
| 正確性 | opus | テストコードのみ |
| セキュリティ | opus | 常に opus（判定に推論余力が必要） |
| 設計+再利用 | opus | ドキュメントのみ |
| 効率 | opus | 常に opus（ホットパス判定が難しい） |

#### エージェント選択

4つ全てを常に起動するのではなく、Phase 1 の結果に基づいて起動するエージェントを選択する。

| 変更内容 | 正確性 | セキュリティ | 設計+再利用 | 効率 |
|---------|--------|------------|------------|------|
| アプリケーションコード（Go / TypeScript 等） | ✅ | ✅ | ✅ | ✅ |
| インフラ（Terraform 等） | ✅ | ✅ | ✅ | - |
| テストコードのみ | ✅ | - | ✅ | - |
| CI/CD 設定のみ | ✅ | ✅ | - | - |
| スキル定義 / ドキュメントのみ | - | - | ✅ | - |

判断に迷う場合は4つ全て起動する(安全側に倒す)。

#### エージェントへの共通入力

以下を各エージェントのプロンプトに含める:

- diff 全文
- 変更の意図(Phase 1a の結果)
- 影響範囲サマリー(Phase 1b の結果)

#### エージェントごとの追加入力

プロンプトサイズを抑えるため、ルールとテスト状況は観点に関連するもののみ渡す。

| 入力 | 正確性 | セキュリティ | 設計+再利用 | 効率 |
|------|--------|------------|------------|------|
| 言語ルール(該当分) | ✅ | - | ✅ | ✅ |
| テスト状況(Phase 1d) | ✅ | - | ✅ | - |

#### エージェントへの共通指示

以下を各エージェントのプロンプト冒頭に含める:

> あなたはコードレビューの専門エージェントです。以下のコンテキストと diff に基づいてレビューしてください。
>
> **重要**: diff の表面を読むだけでなく、データフロー・制御フロー・状態遷移を追跡して判断すること。変更されたコードの呼び出し元・呼び出し先を辿り、実際のコードパスで問題が発生するかを検証すること。推測ではなく、コードを読んで得た根拠(evidence)を指摘に含めること。
>
> **精度基準**: 指摘する際は、問題が発生する具体的な条件(入力値、コードパス、状態)を示すこと。「〜の可能性がある」だけでは不十分。「〜の場合に〜が呼ばれ、〜が発生する」のように具体的に書くこと。
>
> **裏取り**: `critical` または `should` の指摘を出す場合は、指摘が参照するコードパスを Read/Grep で実際に確認し、evidence に確認結果を含めること。コードを読まずに推測だけで出した指摘は除外される。
>
> **思考方針**: 各指摘について、表面的なパターンマッチで判定せず、拡張思考(think harder / ultrathink)で以下を順に検証すること。
> 1. **新規の問題か** — 変更前のコードでも同じ問題が存在したか。存在したなら変更と無関係なので除外
> 2. **意図的な設計判断でないか** — 変更意図や PR description に照らして、作者が承知の上で選んだ設計ではないか
> 3. **発火条件を具体に書けるか** — どの入力・状態のときに、どの行が実行され、何が発生するか。これを1文で書けない指摘は除外
> 4. **プロジェクトルール根拠があるか** — プロジェクトの規約(`.claude/rules/` / `AGENTS.md` / 関連 skill 等)のどの規定に抵触するか。汎用ベストプラクティスのみを根拠とする指摘は `nits` 以下に落とす

#### エージェントの共通出力フォーマット

各エージェントは以下のフォーマットで指摘候補を返す:

```markdown
## 指摘候補

- file: path/to/file.go
  line: 42                         # diff 変更後の行番号
  severity: critical | should | nits | ask
  summary: 一行要約
  detail: 問題の説明と修正案
  evidence:
    - 該当コード: path/to/file.go:L42 の `<symbol>` 内 — 該当行の短い引用("...")
    - 呼び出し元/参照元: path/to/caller.go:L17(関数名) など、辿った経路
    - 発火条件: <入力・状態の具体>(この行は省略不可。書けない場合は指摘を出さない)
    - ルール根拠: プロジェクト規約の該当節(ファイル:節)、または "汎用ベストプラクティス(根拠なし)"
```

**evidence フォーマットの必須項目**:
- `該当コード` と `発火条件` は必須。これらが欠落した指摘は Phase 2.5 で自動的に `unconfirmed` 扱いとなり除外される
- `呼び出し元/参照元` は `critical` / `should` の指摘で必須。`nits` / `ask` では任意
- `ルール根拠` が "汎用ベストプラクティス(根拠なし)" の場合、`nits` を超える severity は付けない

指摘がない場合は「指摘候補なし」と返す。

#### エージェント 1: 正確性レビュアー

**担当**: コードが正しく動くか

プロンプトに含める観点:

- ロジックにバグや抜け漏れがないか
- エッジケース(nil/null、空コレクション、境界値)の処理
- 並行処理の安全性(race condition、TOCTOU)
- リソースリーク
- エラーハンドリングの漏れ

#### エージェント 2: セキュリティレビュアー

**担当**: セキュリティ上の脆弱性

プロンプトに含める観点:

- インジェクション(SQL、コマンド、XSS、テンプレートインジェクション)
- 認証・認可の漏れ
- シークレットの漏洩
- 入力バリデーション
- SSRF、パストラバーサル

#### エージェント 3: 設計+再利用レビュアー

**担当**: コードの構造、一貫性、および既存コードとの再利用性

プロンプトに含める観点:

- 責務の分離が適切か(1つの関数/クラスが多くのことをしていないか)
- 既存パターンとの一貫性(プロジェクトルールに基づく)
- dead code、未使用の import/変数
- 命名の適切さ
- テストの妥当性(変更に対応するテストがあるか)
- **既存ユーティリティ・ヘルパーとの重複**: 新規コードが既存の関数・モジュールで代替可能でないか。ユーティリティディレクトリ、共有モジュール、隣接ファイルを検索して確認する
- **コピー&ペーストの変種**: 微妙に異なる重複コードブロックが共通化できないか
- **不要なコメント**: コードが自明に表現している WHAT コメント、変更履歴コメント、タスク参照コメントは削除対象。非自明な WHY コメント(隠れた制約、ワークアラウンド)のみ残す

#### エージェント 4: 効率レビュアー

**担当**: パフォーマンスとリソース効率

プロンプトに含める観点:

- **不要な処理**: 冗長な計算、重複するファイル読み取り、重複するネットワーク/API 呼び出し、N+1 クエリ
- **並行化の見落とし**: 独立した処理が逐次実行されているが並列実行可能なケース
- **ホットパスの肥大化**: 起動時やリクエスト/レンダーごとに実行されるホットパスに新しいブロッキング処理が追加されていないか
- **無駄な更新**: ポーリングループ、インターバル、イベントハンドラ内で変更検出なしに無条件で state/store 更新していないか
- **メモリ**: 無制限に成長するデータ構造、クリーンアップ漏れ、イベントリスナーリーク
- **過度に広い操作**: 一部だけ必要なのにファイル全体を読み込む、1件だけ必要なのに全件ロードする等

### Phase 2.5: Validation(指摘の検証)

Phase 2 の指摘候補のうち、`critical` または `should` の指摘に対して検証を行う。
`nits` と `ask` は検証不要(影響が小さい / 質問であるため)。

#### 検証の責務分担

各エージェントは共通指示の「裏取り」ルールに従って自己検証済み。メインエージェントはその evidence の妥当性を確認し、以下のいずれかに分類する:

| 分類 | 基準 | 扱い |
|------|------|------|
| confirmed | evidence にコードの具体的な引用・行番号があり、発火条件が具体的 | Phase 3 へ進む |
| likely | evidence に根拠はあるが一部未確認の前提がある | Phase 3 へ進む(未確認の前提を明記) |
| unconfirmed | evidence が曖昧、`該当コード` または `発火条件` が欠落、または推測のみ | 除外 |

#### 実行方法

- **インラインモード**: メインエージェントが Phase 2 と同一コンテキストで直接検証する。`critical`/`should` の指摘については**拡張思考(think harder)で evidence を再構成**し、発火コードパスを頭の中で実行する。evidence が自力で再構成できない指摘は `unconfirmed` として除外
- **並列エージェントモード**: メインが evidence の妥当性を確認する。`likely` 未満で裏が取れない場合は Read/Grep で追加確認し、それでも取れなければ除外

### Phase 3: フィルタリング(指摘の精査)

Phase 2.5 で confirmed/likely と分類された指摘候補を、Phase 1 のコンテキストに照らして精査する。**このフェーズが最も重要。**

#### 個別指摘の精査

各指摘候補に対して:

1. **本当に問題か？** — 変更の意図・背景を考慮しても問題と言えるか
2. **影響範囲は正しいか？** — 呼び出し元を確認した上での指摘か、推測か
3. **プロジェクトルールに基づいているか？** — 汎用的なベストプラクティスではなく、このプロジェクトのルールに照らして問題か
4. **actionable か？** — 具体的な修正方法を示せるか

以下に該当する指摘は**除外する**:

- diff の変更と無関係な既存コードへの指摘(変更に触発されただけの指摘)
- プロジェクト規約に明記されたパターンに従っているコードへの指摘
- 変更の意図を理解すれば自明な設計判断への `ask`
- lint / formatter が検出するスタイルの問題(ツールに任せる)
- レビュー対象外ファイル(後述の「レビュー対象外ファイル」節を参照)への指摘
- **過剰な一般的ベストプラクティス**: プロジェクト規約に根拠がなく、かつプロジェクト内の類似コードで同様のパターンが許容されている場合は除外(強力な推論ゆえに一般論を展開しがちなので意識的に切る)
- **発火条件が書けない指摘**: evidence に「どの入力でどの行が実行されると問題が起きるか」が具体的に書かれていない指摘は除外(Phase 2.5 で `unconfirmed` に落ちているはずの最終チェック)
- **自己矛盾する指摘**: エージェント A の指摘とエージェント B の指摘が両立しない場合(例: 「ここで early return すべき」vs「ここで処理を継続すべき」)、メインが evidence を比較して妥当な方だけ採用し、反対側は除外

#### 複合問題の検出

異なるエージェント間の指摘が交差する箇所を検出する:

- 同じファイル・行に複数エージェントから指摘 → 複合的な問題の可能性
- エージェント A が「エッジケースあり」+ エージェント B が「入力バリデーション不足」→ 組み合わせると脆弱性になりうる
- 個別には `should` でも、複合すると `critical` に昇格するケースがある

## 出力フォーマット

### タグ体系（4タグ）

レビューコメントには以下の4タグを使用する。上から順に判定し、最初に該当したタグを付与する（Decision Tree）。

| # | タグ | 判定基準 |
|---|------|---------|
| 1 | `critical` | コードが壊れる。常に、または**特定条件下**でクラッシュ、データ破損、セキュリティ侵害が発生する。条件がある場合は具体的に示すこと |
| 2 | `should` | 動作するが設計・品質に問題がある。エラーハンドリング欠如、DRY 違反、パフォーマンス問題 |
| 3 | `nits` | スタイル、命名、ドキュメントの正確性。修正は任意 |
| 4 | `ask` | 意図を確認したい。正しさが文脈に依存する |

### レビュー対象外ファイル

以下の自動生成・ベンダリングされたファイルにはレビューコメントを付けない:

- lock ファイル(`package-lock.json`, `pnpm-lock.yaml`, `go.sum`, `Cargo.lock`, `poetry.lock` 等)
- コード生成ツールの出力(GraphQL codegen、protobuf、ORM 生成コード、`*_mock.go` 等)
- ベンダリングされた依存(`vendor/`, `node_modules/` 等)

判断に迷う場合はプロジェクトの慣習(`.gitattributes` の `linguist-generated` 指定等)に従う。

### ルール参照ポリシー

- プロジェクト規約に記載されている事項への `ask`(意図確認)は不要。規約を確認すれば分かるため
- ルール違反は `should` 以上で指摘する

### バッジの使い分け

出力先に応じてバッジ形式を切り替える:

| 出力先 | フォーマット | 例 |
|--------|------------|-----|
| GitHub 投稿 | 画像バッジ | `![critical](https://img.shields.io/badge/review-critical-red.svg)` |
| ターミナル表示(ローカル直接呼び出し) | テキストバッジ | `[critical]`, `[should]`, `[nits]`, `[ask]` |

### 指摘の書き方

各指摘は以下を含むこと:

- **タグ**(severity badge)
- **ファイルパスと行番号**
- **問題の説明**(何が問題で、どの条件で発生するか)
- **根拠**(どのコードパスで問題が発生するか、呼び出し元・呼び出し先の追跡結果)
- **修正案**(具体的な修正方法またはコード例)
- **対応判定**(以下のいずれか)
  - **対応推奨** — 指摘が妥当で修正すべき。理由を簡潔に付記
  - **対応不要** — 意図的な設計、既知の制約、または影響が軽微。理由を簡潔に付記
  - **要確認** — 正しさが文脈に依存し、PR 作成者の判断が必要(`ask` タグと組み合わせて使用)

ターミナル表示の例:

```markdown
[critical] `internal/handler/user.go:42`

`userID` が空文字列の場合、`strings.Split` が空スライスを返し、
`parts[0]` で index out of range パニックが発生する。

呼び出し元の `handleRequest`(L28)では空文字列チェックがないため、
外部入力がそのまま到達する。

```go
if userID == "" {
    return failure.New("userID is empty")
}
```
```

### サマリー

レビュー結果は以下の構造で出力する:

```markdown
## レビューサマリー

**変更の意図**: {Phase 1a で把握した変更の目的}
**影響範囲**: {Phase 1b で調査した影響範囲の要約}

### 指摘一覧

| # | ファイル:行 | タグ | 概要 | 対応 |
|---|-----------|------|------|------|
| 1 | path/to/file.go:42 | critical | 説明 | **対応推奨** — 理由 |

### 判定: {APPROVE / REQUEST_CHANGES / COMMENT}(参考。GitHub 投稿時の event とは独立)
```

### 指摘がゼロの場合

Phase 3 のフィルタリングで全ての指摘候補が除外された場合:

- 判定は **APPROVE** とする
- 「指摘なし」であることを明示し、Phase 1 のコンテキスト収集結果(変更の意図・影響範囲)のみ出力する
- GitHub 投稿が不要であれば投稿しない

## GitHub 投稿(PR 指定時)

### 投稿確認

GitHub にレビューを投稿する場合、投稿前に必ずユーザーに確認を取る:

1. レビュー結果のサマリーと全指摘を提示する
2. 判定(APPROVE / REQUEST_CHANGES / COMMENT)を明示する
3. ユーザーが承認 → 投稿
4. ユーザーが修正を指示 → コメントを修正して再提示

**ユーザーの確認なしに GitHub へ投稿しない。**

### デフォルト event

- GitHub への投稿時、デフォルトの event は **COMMENT** とする
- ユーザーが明示的に指示した場合のみ `REQUEST_CHANGES` / `APPROVE` を使う
- ターミナル提示時の判定(APPROVE / REQUEST_CHANGES / COMMENT)は参考情報として出力するが、実際の投稿 event とは独立

### 投稿フォーマット

```bash
gh api repos/:owner/:repo/pulls/$PR_NUMBER/reviews \
  --method POST --input - <<'EOF'
{
  "body": "![AI Generated](https://img.shields.io/badge/AI-Generated-blueviolet)\n\nレビューサマリー",
  "event": "COMMENT",
  "comments": [
    {
      "path": "internal/handler/user.go",
      "line": 42,
      "body": "![critical](https://img.shields.io/badge/review-critical-red.svg)\n\nコメント本文"
    }
  ]
}
EOF
```

- レビュー本文には `![AI Generated]` バッジを付与
- 各インラインコメントには種別バッジ(critical/should/nits/ask)を付与
- `line` は diff の変更後の行番号を指定する(`side: "RIGHT"` がデフォルト)
- 複数行にまたがる場合は `start_line` と `line` で範囲指定可能

## レビューコンテキスト別の注意

| コンテキスト | 影響範囲の調査 | 投稿確認 | バッジ |
|---|---|---|---|
| CI 環境(自動レビュー) | Grep/Read | 不要(自動投稿) | 画像バッジ |
| ローカル(PR 指定) | コードインデックス MCP 優先、無ければ Grep/Read | 必要 | 画像バッジ |
| ローカル(直接呼び出し) | MCP or Grep | 投稿時のみ | テキストバッジ |

## レビュー後のアクション

Phase 3 の出力後、呼び出しコンテキストに応じてアクションを分岐する。

### 呼び出し元による分岐

| コンテキスト | 判定方法 | アクション |
|------------|---------|----------|
| CI 環境 | CI 環境変数が設定されている | 自動投稿(確認不要) |
| author = 現在のユーザー | PR author が現在のユーザー、またはローカルブランチの変更 | レビュー結果を提示 → 「指摘に基づいて修正しますか？」と確認 → 承認されたら修正を実行 |
| author ≠ 現在のユーザー | PR author が現在のユーザー以外 | レビュー結果を提示 → 「GitHub にレビューを投稿しますか？」と確認 → 承認されたら投稿 |

### author のコードの修正フロー

1. レビュー結果のサマリーと全指摘を提示する
2. 「以下の指摘に基づいて修正しますか？ 修正対象を選択できます(all / 番号指定 / none)」と確認
3. ユーザーが選択 → 選択された指摘に対して修正を実行
4. 修正完了後、変更内容のサマリーを提示

## 例

```bash
/review-patch 5612
/review-patch https://github.com/owner/repo/pull/5612
/review-patch
/review-patch feature/foo
```
