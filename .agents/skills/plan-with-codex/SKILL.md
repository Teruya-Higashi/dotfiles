# Plan 生成スキル（with Codex）

サブエージェント（Agent ツール）と Codex（`codex exec` CLI）の2系統で同じタスクの修正計画を並列で作成させ、
両方の結果をマージした統合 plan ファイルを出力するスキル。
plan ファイルは後続の修正作業（Claude Code 等）の input として使う。

Codex チャネルは plugin（`codex:codex-rescue`）を経由せず、`codex` バイナリを Bash から直接呼び出す。
認証・サンドボックス設定は `~/.codex/config.toml` に従う（`approval_policy = "never"` を前提）。
モデル・effort は CLI フラグで明示し、config の値を上書きする。

## 前提

- `command -v codex` で Codex CLI が見つかること
- `gpt-5.6-sol` を使うため Codex CLI `0.144.0` 以降であること（`codex --version` で確認）
- 古い CLI では暗黙に旧モデルへフォールバックせず、更新を案内する

## 引数

```
/plan-with-codex [タスクの説明・要件・コンテキスト]
```

- **タスクの説明**: Plan 生成に必要な情報（必須）

## オプション

引数内に以下のキーワードが含まれる場合、対応するパラメータを切り替える。

| オプション | デフォルト | 引数内キーワード例 |
|---|---|---|
| 出力プレフィックス | `plan` | `--prefix my-plan` |
| Codex モデル | `gpt-5.6-sol` | `--model gpt-5.4` |
| Codex effort | `xhigh` | `--effort high`, `--effort max` |

**パース規則**:
- `--prefix {値}`, `--model {値}`, `--effort {値}` を引数から抽出し、残りをタスクの説明として扱う
- キーワードが見つからなければデフォルト値（`--model gpt-5.6-sol --effort xhigh`）を使用

**Codex CLI へのマッピング**:
- `--model {値}` → `codex exec` の `-m {値}`（指定値をそのまま渡す。未指定時は `gpt-5.6-sol`）
- `--effort {値}` → `-c model_reasoning_effort="{値}"`（未指定時は `xhigh`）
- plan ファイルへの書き出しは `-o {ファイル}`（エージェントの最終メッセージをファイルに書き出す）

### effort の使い分け

- `xhigh`（デフォルト）: 複数ファイル・複数レイヤーの調査を含む通常の計画作成
- `high`: 対象範囲が狭く、既存パターンが明確な計画で反復速度を優先する場合
- `max`: 大規模移行、複雑なデータフロー、セキュリティ境界を含む計画など、最深の単一実行推論が必要な場合

```text
/plan-with-codex --effort high 小規模な設定変更の修正計画を作る
/plan-with-codex --effort max 複数サービスにまたがる認証基盤移行の修正計画を作る
```

GPT-5.6 Sol は `ultra` も利用できるが、自動タスク委任により固定の2系統構成が変わるため、このスキルでは使わない。深い推論が必要な場合は `max` を使う。ユーザー指定なしに effort を自動変更しない。

出力ファイル名は `{prefix}-{ai_name}_{sequence_number}.md` と `{prefix}-codex_{sequence_number}.md`

**`{ai_name}` の決定**: 現在スキルを実行している AI の名前を使用する（例: `claude`, `gemini` 等）。自身のモデル名から判定すること。

## Plan ファイルのフォーマット

サブエージェント・Codex 両方に以下のフォーマットで出力させる。

````markdown
# 修正計画: {タスクの要約}

## 実現したいこと

[この変更で達成するゴールを1-2文で簡潔に]

## 背景

[なぜこの変更が必要か。現状の課題・動機・経緯を記述]

## 要件

1. [要件1]
2. [要件2]
3. ...

## 設計

### 既存実装の分析

[関連する既存パターンのトレース。テーブル形式で流れを可視化]

| フェーズ | ファイル | 処理 |
|---|---|---|
| ... | `path/to/file.go` | 処理の説明 |

### 調査済みの事実・推測・未解決事項

- **確認済み**: [ファイル・シンボル・設定等の根拠を添える]
- **推測**: [推測である理由と検証方法を添える]
- **未解決**: [実装前に確認が必要な判断事項。なければ「なし」]

### 方針

[アプローチの説明。既存パターンとの差分がある場合はテーブルで比較]

| 観点 | 既存パターン | 今回の方針 |
|---|---|---|
| ... | ... | ... |

### データフロー / 構造

[必要に応じて、データの流れやコンポーネント間の関係を図示・説明]

## 実装

具体的な変更内容をサンプルコードを交えて記述する。
カテゴリごとにセクションを分ける。

### カテゴリ名（例: ヘッダー定義、DB モデル等）

**`path/to/file.go`**

[変更内容の説明]

```go
// 変更前
existing code...

// 変更後
modified code...
```

**`path/to/file2.go`**

```go
// サンプル
new code...
```

## 修正チェックリスト

修正対象をカテゴリ別にチェックリストで記載。
各項目にはファイルパスと具体的な変更内容を含める。

### コア変更

- [ ] `path/to/file.go` — 変更内容の説明
- [ ] `path/to/file2.go` — 変更内容の説明

### テスト

- [ ] `path/to/test.go` — テスト内容の説明

### 検証

- [ ] `mise run go:lint` で lint 通過
- [ ] `mise run go:test` で全テスト通過
````

## GPT-5.6 Sol 向けの Plan 方針

Codex チャネルには完成形を先回りして教えず、目的・制約・受け入れ条件を渡して、長いコンテキストと多段階のツール利用をリポジトリ調査に使わせる。

- `AGENTS.md`、README、ビルド・テスト設定、関連コードを Codex 自身に確認させる
- 変更点だけでなく、エントリーポイントからデータ・制御の流れ、下流の利用箇所、テストまでを追跡させる
- ファイルパスと可能ならシンボル名を根拠として示し、存在しない API や抽象化を発明させない
- 確認済みの事実、推測、未解決事項を区別させる
- 実装順序は依存関係と安全な検証単位に基づかせ、単にファイル一覧を並べた plan にしない
- 互換性、データ移行、セキュリティ、ロールバックが関係する場合だけ、それぞれの具体的な対応を含めさせる
- 受け入れ条件と実行可能な検証コマンドを含めさせる
- Plan 生成中はソースコードを変更させない

## 手順

### 1. 入力の確認

引数からタスクの説明とオプションを取得する。引数が空の場合は AskUserQuestion でタスク内容を確認する。

### 2. 連番の決定

既存の plan ファイルを確認し、次の連番を決定する:

```bash
ls {prefix}-{ai_name}_*.md {prefix}-codex_*.md 2>/dev/null | sort -V | tail -1
```

ファイルが存在しなければ連番は `1` から開始する。

### 3. 並列 Plan 生成の実行

サブエージェント（Agent ツール）と Codex（`codex exec` CLI）を**同時にバックグラウンドで**起動する。
必ず同一ターン内で両方の呼び出しを行うこと（Agent ツールと Bash ツールを1ターンで並行起動する）。

#### a. サブエージェント Plan 生成（Agent ツール）

```
Agent(
  subagent_type: "general-purpose",
  mode: "bypassPermissions",
  run_in_background: true,
  prompt: "シニアエンジニアとして以下のタスクの修正計画を作成し、{prefix}-{ai_name}_{sequence_number}.md に書き出してください。

  タスク: {タスクの説明}

  Plan ファイルのフォーマット: {上記フォーマット}

  指示:
  - 既存コードベースを調査し、既存パターンをトレースすること
  - エントリーポイントから下流の利用箇所・テストまで影響範囲を追跡すること
  - 修正対象ファイルはパスと変更内容を具体的に記載
  - 修正チェックリストは漏れなく列挙
  - 設計判断の根拠を明記
  - 確認済みの事実、推測、未解決事項を区別すること
  - 受け入れ条件と実行可能な検証コマンドを含めること
  - ソースコードは変更せず、Plan のみを出力すること
  "
)
```

#### b. Codex Plan 生成（`codex exec` CLI）

`codex exec` を Bash ツールで `run_in_background: true` 起動し、Codex に Plan 生成を委任する。
`-o {ファイル}` でエージェントの最終メッセージを plan ファイルに直接書き出す。
プロンプトは多行のため heredoc で stdin から渡す。

```bash
codex exec -m {model} -c model_reasoning_effort="{effort}" -o {prefix}-codex_{sequence_number}.md - <<'EOF'
シニアエンジニアとして以下のタスクの修正計画を作成してください。
関連ファイルの調査・既存パターンのトレースもあなた自身で行うこと。

タスク: {タスクの説明}

Plan ファイルのフォーマット: {上記フォーマット}

指示:
- 既存コードベースを調査し、既存パターンをトレースすること
- AGENTS.md、README、ビルド・テスト設定を確認し、リポジトリ固有の指示に従うこと
- エントリーポイントからデータ・制御の流れ、下流の利用箇所、テストまで影響範囲を追跡すること
- 修正対象ファイルはパスと変更内容を具体的に記載
- 修正チェックリストは漏れなく列挙
- 設計判断の根拠を明記
- 確認済みの事実、推測、未解決事項を明確に区別し、根拠となるファイル・シンボルを示すこと
- 存在しない API や抽象化を発明せず、既存実装に根ざした計画にすること
- 受け入れ条件と実行可能な検証コマンドを含めること
- ソースコードは変更せず、Plan のみを出力すること

重要: あなたの最終応答（last message）として plan の全文を上記フォーマットで出力すること。
要約や中間報告ではなく、全セクションを省略せず最終メッセージに含めること。
EOF
```

**CLI フラグの構築**:
- `-m {値}` → 常に付与（指定値をそのまま渡す。未指定時は `gpt-5.6-sol`）
- `-c model_reasoning_effort="{値}"` → 常に付与（未指定時は `xhigh`）
- `-o {prefix}-codex_{sequence_number}.md` → 常に付与（plan ファイル書き出しのため）
- サンドボックス・承認は `~/.codex/config.toml` に従う（`workspace-write` / `never` 前提）

### 4. 完了待ち

両方のバックグラウンドタスクの完了通知を受け取る。
ユーザーから途中経過の確認があった場合は `TaskOutput`（`block: false`）で状態を報告する。

進捗テーブルを表示して状況を共有する:

```markdown
| タスク | 状態 |
|---|---|
| {ai_name} Plan → `{prefix}-{ai_name}_{seq}.md` | 完了 / 実行中 |
| Codex Plan → `{prefix}-codex_{seq}.md` | 完了 / 実行中 |
| マージ | 待機 |
```

### 5. Plan のマージ・統合

両方の plan ファイルを Read ツールで読み込み、以下の方針でマージする:

1. **両者が共通で挙げた修正対象**: 高確信度として採用
2. **片方のみの修正対象**: コードベースを確認し、妥当性を判断
3. **矛盾する方針**: 両方の根拠を比較し、より妥当な方を採用（判断理由を明記）
4. **重複の統合**: 同一ファイルへの修正チェックリスト項目を1つにまとめる

**統合 plan のフォーマット**: サブエージェント・Codex の出力には背景・要件・設計を含むが、統合 plan からは分析セクション（背景・要件・設計の既存実装分析・補足）を除去し、「実現したいこと」「実装」「修正チェックリスト」のみで構成する。分析はマージ判断の材料として使うが、最終出力には残さない。

### 6. 統合 Plan の提示とユーザー確認

マージした plan を `{prefix}_{sequence_number}.md` に書き出す。

サブエージェント/Codex の分析の違いを差分テーブルで可視化する:

```markdown
| # | 項目 | {ai_name} | Codex | 統合判断 |
|---|------|-------|-------|----------|
| 1 | 修正対象 A | あり | あり | 採用（合意） |
| 2 | 修正対象 B | あり | なし | 採用 — 理由 |
| 3 | 方針 C | 方法X | 方法Y | 方法X — 理由 |
```

AskUserQuestion でユーザーに確認を取る:
- plan の内容に問題がないか
- 追加・修正したい項目があるか

**重要**: ユーザーの確認なしに plan を確定しない。

### 7. 結果の報告

最終的な plan ファイルのパスを報告する:

```
統合 Plan: {prefix}_{sequence_number}.md
参考: {prefix}-{ai_name}_{sequence_number}.md / {prefix}-codex_{sequence_number}.md
```

## 注意事項

- サブエージェントと Codex は必ず**並列で**起動すること（同一ターン内で両方の呼び出しを行う）
- Codex への依頼は `codex exec` CLI を Bash から直接呼び出す。plugin（`codex:codex-rescue`）や `codex-companion.mjs` は経由しない
- Codex へのタスク依頼は事前に情報を収集・加工せず、関連ファイルの調査・既存パターンのトレースも含めて Codex 自身に行わせる
- GPT-5.6 Sol の長時間調査を活かし、単に対象範囲が広いという理由で調査を途中で打ち切らない。独立した成果物やユーザー判断が必要な境界だけを分割する
- 統合 plan はユーザーに提示して確認を取る
- 両モデルの方針が矛盾する場合は、コードの文脈を確認した上で判断理由を明記する
- plan ファイルのフォーマットは修正チェックリスト形式を基本とする
