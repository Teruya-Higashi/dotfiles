# Codex タスク委任スキル

Codex（OpenAI のコーディングエージェント）に `codex exec` CLI コマンドでタスクを委任するスキル。

## 引数

```
/codex [タスクの説明]
```

- **タスクの説明**: Codex に実行させたいタスクの内容（必須）

## オプション

引数内に以下のキーワードが含まれる場合、対応するパラメータを切り替える。
指定がなければデフォルト値を使用する。

| オプション | デフォルト | 選択肢 | 引数内キーワード例 |
|---|---|---|---|
| モデル | `gpt-5.4` | `gpt-5.4`, `gpt-5.3-codex`, `gpt-5.2-codex`, `o3` 等 | `--model o3` |
| reasoning_effort | `xhigh` | `low`, `medium`, `high`, `xhigh` | `--reasoning high` |
| 出力ファイル | なし | 任意のファイルパス | `--output ./review.md` |

**パース規則**:
- `--model {値}`, `--reasoning {値}`, `--output {値}` を引数から抽出し、残りをタスクの説明として扱う
- キーワードが見つからなければデフォルトを使用

**出力ファイルの扱い**:
- `--output` が指定された場合、`-o` フラグは使わず、プロンプト内にファイル書き出し指示を埋め込む
- 理由: `-o` フラグは Codex の最終アシスタントメッセージを書き出す仕様のため、Codex がファイル操作で内容を書き込んだ後に要約メッセージを返すと、要約で上書きされてしまう

## 手順

### 1. タスク内容の確認

引数からタスクの説明とオプションを取得する。引数が空の場合は AskUserQuestion でタスク内容を確認する。

### 2. Codex セッションの開始

Bash ツールで `codex exec` コマンドをバックグラウンド実行する。

**`--output` 指定なしの場合**:

```bash
codex exec --sandbox workspace-write -m {モデル} -c model_reasoning_effort={reasoning_effort} "{タスクの説明}"
```

**`--output` 指定ありの場合**:

`-o` フラグは使わず、プロンプト末尾にファイル書き出し指示を追加する:

```bash
codex exec --sandbox workspace-write -m {モデル} -c model_reasoning_effort={reasoning_effort} "{タスクの説明}

重要: 結果の全文を {出力ファイル} にファイル操作で直接書き出してください。要約ではなく、全セクション・全指摘を省略せずに記載すること。"
```

**パラメータの説明**:
- `--sandbox workspace-write`: プロジェクトディレクトリへの書き込みを許可
- `-m {モデル}`: 使用するモデル（デフォルト: `gpt-5.3-codex`）
- `-c model_reasoning_effort={reasoning_effort}`: 推論エフォート（デフォルト: `xhigh`）

**実行オプション**:
- `run_in_background: true` でバックグラウンド実行する
- `timeout: 600000`（10分）を設定する

### 3. 結果の確認

バックグラウンドタスクの完了通知を受け取ったら:
- `TaskOutput` で結果を確認する
- 成功した場合: 変更内容のサマリーをユーザーに報告
- 失敗した場合: エラー内容を報告し、次のアクションを提案
- ユーザーから途中経過の確認があった場合: `TaskOutput`（`block: false`）で現在の状態を確認

### 4. フォローアップ（必要に応じて）

`codex exec` はステートレスなので、追加指示がある場合は新しい `codex exec` コマンドで実行する。
前回の結果を踏まえた指示をプロンプトに含める。

## 注意事項

- ユーザーの依頼内容を変えずにそのまま Codex に委任すること。事前に情報を収集・加工して prompt に含めない
  - 例: レビュー依頼の場合、差分の取得も Codex 自身に行わせる（呼び出し側で `git diff` して結果を渡さない）
  - 例: バグ調査の場合、関連ファイルの読み取りも Codex に任せる（呼び出し側で事前にファイルを読まない）
- Codex の変更内容は `git diff` で確認し、意図しない変更がないかレビューする
- 大規模な変更を依頼する場合は、タスクを分割して段階的に実行する
- MCP 経由（`mcp__codex__codex`）は長時間ハングする場合があるため、CLI 直接実行を優先する
