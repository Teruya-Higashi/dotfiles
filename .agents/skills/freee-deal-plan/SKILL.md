# freee deal-plan.md 作成スキル

## 概要

このスキルは freee の取引未登録証憑を対象に、既存取引パターンを参照して新規取引ドラフトを推定し、`deal-plan.md` に整理して出力する。

## 適用条件

- 対象は `category=without_deal` の証憑。
- 期間は `start_date` と `end_date` を明示して取得する。
- 事業所は current company と `company_id` を一致させる。

## 厳守事項

- 取引登録禁止: `POST /api/1/deals` を実行しない。
- 既存取引への紐付け禁止: 証憑更新APIで取引IDを設定しない。
- 変更系API禁止: `PUT/PATCH/DELETE` は使わない（GETのみ）。

## 入力

- `company_id`
- `start_date`（YYYY-MM-DD）
- `end_date`（YYYY-MM-DD）
- `output_file`（通常 `deal-plan.md`）

## 中間ファイル設計

コンテキストウィンドウの消費を抑えるため、各Phaseの結果は中間ファイルに保存し、後続Phaseはファイルを読み取って処理する。APIレスポンスの生データをメインコンテキストに展開しない。

### ディレクトリ

`{output_file の親ディレクトリ}/.intermediate/` に保存する（例: `deal-plan.md` が出力先なら `.intermediate/`）。

### ファイル一覧

| Phase | ファイル | 内容 |
|---|---|---|
| 2 | `masters.json` | マスタデータ（`{カテゴリ: {ID: ラベル}}` 形式に縮約） |
| 3 | `receipt_ids.txt` | 証憑ID一覧（1行1ID、機械カウント用） |
| 3 | `receipts_raw.json` | 証憑一覧APIの生レスポンス（OCRメタデータ含む） |
| 3 | `receipts_batch_{n}.json` | 証憑読み取り結果（バッチ別、サブエージェント出力） |
| 3 | `receipts.json` | 証憑読み取り結果（全バッチ結合済み） |
| 4 | `patterns.json` | 登録済み証憑ID一覧 + 既存取引パターン |

### サブエージェント運用ルール

- Phase 2〜4 は各々 Task（サブエージェント）で実行する。
- サブエージェントは結果を中間ファイルに Write し、メインエージェントにはサマリー（件数等）のみ返す。
- メインエージェントは中間ファイルのパスとサマリーだけを保持し、大量データを直接コンテキストに載せない。

## 実施手順

### Phase 1: 認証・事業所確認

1. `freee_auth_status` と `freee_get_current_company` で認証・事業所を確認する。
2. **認証が無効な場合のリカバリー**:
   - ユーザーに認証が無効であることを通知する。
   - `npx @him0/freee-mcp configure` を Bash で実行する。
   - コマンドの入出力内容をユーザーに提示し、**AskUserQuestion** で確認を仰ぐ。
   - 認証完了後、再度 `freee_auth_status` で有効性を検証してから次のPhaseに進む。

### Phase 2 & 3a: マスタデータ取得 + 証憑一覧取得（並列実行）

Phase 2（マスタデータ取得）と Step 3a（証憑一覧取得）は互いに依存しないため、**並列実行**する。

#### Phase 2: マスタデータ一括取得（サブエージェント）

2. **サブエージェント**で以下のマスタデータを一括取得し、`{ID: ラベル}` マッピングのみ抽出して `.intermediate/masters.json` に保存する。
   - 取引先: `GET /api/1/partners` → `{"partners": {"10000001": "サンプル株式会社", ...}}`
   - 勘定科目: `GET /api/1/account_items` → `{"account_items": {"20000001": "消耗品費", ...}}`
   - 税区分: `GET /api/1/taxes/codes` → `{"tax_codes": {"21": "課対仕入10%", ...}}`
   - メモタグ: `GET /api/1/tags` → `{"tags": {"30001": "サンプルタグ", ...}}`
   - 品目: `GET /api/1/items` → `{"items": {"40001": "サンプル品目", ...}}`
   - 決済口座: `GET /api/1/walletables` → `{"walletables": {"bank_account:50001": "〇〇銀行", ...}}`
   - サブエージェントはメインに各カテゴリの件数サマリーのみ返す。

#### Step 3a: 証憑一覧取得と件数カウント（メインエージェントまたは別サブエージェント）

3. **Phase 2 と並列に**、`GET /api/1/receipts` を `category=without_deal` で取得する。ページネーションがある場合は全ページ取得する。
4. レスポンスから `receipt_id` を1行1件で `.intermediate/receipt_ids.txt` に Write する。
5. **Bash で `wc -l < .intermediate/receipt_ids.txt` を実行して件数をカウントする。**
   - **LLMが配列を目視で数えることは厳禁。** 必ず `wc -l` の結果を正とする。
   - `meta.total_count` がレスポンスにあれば `wc -l` の結果と一致することを検証する。

#### Step 3b: ユーザー確認ゲート（メインエージェント）

6. **AskUserQuestion** で以下をユーザーに提示し、承認を得てから後続処理に進む。承認なしで Phase 3c 以降に進むことは禁止。
   - 未登録証憑の件数
   - 内訳（PDF / PNG / その他の件数）
   - 証憑の作成日の範囲（最古〜最新）
   - 「この件数で処理を続行してよいか？」

#### Step 3c: 証憑ダウンロード・読み取り（サブエージェント × 最大3）

7. 各証憑の内容（取引先・日付・金額・概要）を特定する。証憑の情報が不足する場合は `GET /api/1/receipts/{id}/download` でファイルをダウンロードし、内容を読み取る。
8. 証憑件数が多い場合は並列サブエージェントでダウンロード・読み取りを分担する。
   - **並列数の制限**: 最大3つ。1エージェントあたり15〜25件を目安（例: 30件→2並列×15件、60件→3並列×20件）。10件未満は逐次処理でもよい。
   - 各サブエージェントは担当分の結果を `.intermediate/receipts_batch_{n}.json` に Write する。
   - フォーマット: `[{"receipt_id": 123, "mime": "pdf", "partner_name": "...", "issue_date": "...", "amount": 1234, "description": "..."}, ...]`
   - **レートリミット対策**: freee APIのレートリミット（3,600回/時）に注意する。並列サブエージェントが同時にダウンロードAPIを呼ぶと429エラーが発生しやすい。エラー時は一定時間待機後にリトライする。全バッチ失敗時はメインエージェントが再起動を判断する。
   - **PNG証憑の取り扱い**: PNG証憑はクレジットカード利用明細のスクリーンショットであることが多い。ダウンロード後、Read ツール（画像読み取り）で内容を確認する。OCRツール（tesseract等）の可用性に依存しないこと。partner_name はカード明細の表記（例: "OPENAI *CHATGPT SUBS"）となるため、取引先名マッチングルールに従い正規化する。
9. メインエージェントが全バッチを結合して `.intermediate/receipts.json` に保存する。
10. **Bash で `jq 'length' .intermediate/receipts.json` を実行し、件数が `wc -l < .intermediate/receipt_ids.txt` と一致することを検証する。**

### Phase 4: 参照取引パターンの収集（サブエージェント）

11. **サブエージェント**で以下を実行し、結果を `.intermediate/patterns.json` に保存する。
    - `category=with_deal` の登録済み証憑を取得し、類似証憑IDの候補とする。
    - `GET /api/1/deals` で収入・支出の既存取引を取得し、勘定科目・税区分・タグ・品目・決済口座のパターンを把握する。
    - サブエージェントはメインに取引パターン数のサマリーのみ返す。

### Phase 5: ドラフト推定と出力（サブエージェント）

12. **サブエージェント**で以下の中間ファイルを読み取り、ドラフト推定と `deal-plan.md` 出力を行う。
    - 入力: `.intermediate/masters.json` + `.intermediate/receipts.json` + `.intermediate/patterns.json`
    - 参照取引をベースに、対象証憑ごとの取引ドラフトを推定。
    - `推測される新規取引(ドラフト)` を API キー形式で記載。
    - `取引内容` をラベル名形式で記載（ドラフトと同じキー構成）。
    - `deal-plan.md` を上書き更新し、品質チェックを実施。

#### 登録済み取引の判定ロジック

Phase 4 で取得した既存取引（`GET /api/1/deals`）と対象証憑を照合し、以下の条件を **すべて** 満たす場合に「取引登録済み」と判定する。

1. 同一 `partner_id` の取引が存在する
2. `issue_date` が証憑の発行日と一致（±数日の許容範囲あり）
3. `amount` が証憑の金額と一致

判定結果は推定一覧テーブルの `取引登録済み` カラムに `[x]` / `[ ]` で反映し、補足セクション「登録済み取引の識別」にも件数と receipt_id 一覧を記載する。

#### 取引先名マッチングルール

証憑の `partner_name` と patterns.json の取引先名は完全一致しないことが多い。以下の揺れパターンを考慮してマッチングする。

| 証憑の名称例 | マッチ先パターン | ルール |
|---|---|---|
| Amazon Services International, LLC. | Amazon.co.jp | Amazon 関連は全て Amazon.co.jp パターンを参照 |
| フリー株式会社 | freee株式会社 | 日本語/英語表記の揺れ |
| OrbStack | Orbstack | 大文字小文字の揺れ |
| 株式会社〇〇 / 〇〇株式会社 | 〇〇 | 法人格の前後・有無 |
| 〇〇(PayPal) | 〇〇 | 決済手段サフィックスを除去して検索 |
| Anker Japan Co.,Ltd | Amazon.co.jp | Amazon マーケットプレイス出品者 |
| 深セン〇〇有限公司 | Amazon.co.jp | Amazon マーケットプレイス出品者 |
| PAYPAL *OREILLYMEDI | 株式会社オライリー・ジャパン | PayPal プレフィックス除去 + サービス名推定 |
| グーグル・ペーメント・ジ | Google | カタカナ/英語の揺れ |
| OPENAI *CHATGPT SUBS | OpenAI | クレカ明細の略称 |
| ORBSTACK PRO | Orbstack | クレカ明細のプラン名付き |
| DROPBOX | Dropbox | 大文字のみ表記 |

マッチしない場合は、証憑の `description` から用途を推測し、類似する既存パターンの勘定科目・税区分を参考にする。

## deal-plan.md 記載仕様

### 必須セクション

1. `# deal-plan.md`
2. メタ情報（作成日時・対象期間・対象件数・事業所ID・実施内容・厳守事項）
3. `## 推定一覧`
4. `## 補足`

### 必須カラム

1. `receipt_id`
2. `mime`
3. `類似登録済み証憑ID`
4. `参照取引ID`
5. `既存取引テンプレート`
6. `推測される新規取引(ドラフト)`
7. `取引内容`
8. `確度`
9. `根拠`
10. `取引登録済み`
11. `証憑紐付け済み`

### カラム記載ルール

- `receipt_id`: 1行1証憑。重複禁止。
- `推測される新規取引(ドラフト)`: APIキー形式（例: `type`, `partner_id`, `account_item_id`, `tax_code`, `tag_ids`, `item_id`, `payment`, `issue_date`, `amount`）。
- `取引内容`: ドラフトと同じキー構成で、コード値ではなくラベル名で記載（後述の対応表に従う）。
- `確度`: `高` / `中高` / `中` / `低`。
- `根拠`: 1-2文で具体的に記載。
- `取引登録済み`: `[x]` = 取引が既に登録済み / `[ ]` = 未登録（新規作成が必要）。判定は「登録済み取引の判定ロジック」に従う。
- `証憑紐付け済み`: 初期値は常に `[ ]`（本スキルでは紐付け操作を行わないため）。

## ドラフト ↔ 取引内容 キー対応表

`推測される新規取引(ドラフト)` と `取引内容` は **同じキーを必ず両方に記載** する。片方にあるキーは必ず他方にも存在すること。

| ドラフト (APIキー) | 取引内容 (ラベル) | 例 |
|---|---|---|
| `type` | `収支区分` | expense → 支出 / income → 収入 |
| `partner_id` | `取引先` | 10000001 → サンプル書店 |
| `account_item_id` | `勘定科目` | 20000001 → 新聞図書費 |
| `tax_code` | `税区分` | 21 → 課対仕入10% / 11 → 課税売上10% |
| `tag_ids` | `メモタグ` | [30001] → [書籍] |
| `item_id` | `品目` | 40001 → 電気料金 |
| `payment` | `決済口座` | private_account_item:60001 → プライベート資金 |
| `issue_date` | `発行日` | 2025-01-15 → 2025-01-15 |
| `amount` | `金額` | 3000 → 3,000 |

- `tag_ids` / `item_id` は該当する場合のみ記載。ただし片方に記載した場合はもう片方にも必ず記載。
- 取引先が未登録の場合: ドラフトは `partner_id=要新規作成`、取引内容は `取引先=新規(会社名)` と記載。

## ラベル化ルール（取引内容）

`取引内容` では、以下のように **ID/コードではなくラベル名** を使う。

- `partner_id` -> `取引先`（例: `取引先=サンプル書店`）
- `account_item_id` -> `勘定科目`（例: `勘定科目=新聞図書費`）
- `tax_code` -> `税区分`（例: `税区分=課対仕入10%`）
- `tag_ids` -> `メモタグ`（例: `メモタグ=[書籍, 開発]`）
- `item_id` -> `品目`（例: `品目=電気料金`）
- 決済口座ID -> `決済口座` ラベル（例: `決済口座=プライベート資金`）
- `issue_date` -> `発行日`（例: `発行日=2025-01-15`）
- `amount` -> `金額`（例: `金額=3,000`）カンマ区切りで記載

不明な場合は空欄にせず `不明(ID:xxxx)` で残す。

## ラベル解決の参照API

- 取引先: `GET /api/1/partners`
- 勘定科目: `GET /api/1/account_items`
- 税区分: `GET /api/1/taxes/codes`
- メモタグ: `GET /api/1/tags`
- 品目: `GET /api/1/items`
- 決済口座: `GET /api/1/walletables`

## 補足セクションの記載ルール

補足には以下の情報を含めること。

### 登録済み取引の識別

`category=without_deal` の証憑でも、実際には取引が登録済みで証憑紐付けのみ未実施のケースがある。以下を補足に明記する。

- **取引登録済み・紐付けのみ必要**: 取引は存在するが証憑が未紐付け。対象の receipt_id 一覧と対応する取引IDを記載。
- **新規取引登録が必要**: 取引自体が未登録。ドラフト通りに新規作成する。

### 同日・同取引先の証憑グルーピング

同日・同取引先で複数の証憑がある場合（例: 同月のSecurifyScan + ABEMA）、1つの取引に複数の証憑ファイルを紐付ける運用がある。該当するケースは補足に以下を記載する。

- 対象の receipt_id のペア/グループ
- 合算後の取引金額
- 紐付け方法（1取引に複数証憑を添付）

### その他の必須記載事項

- 取引先未登録で新規作成が必要なケースの一覧
- 外貨建てサービス（為替変動による金額差異）の説明
- ラベル未解決件数
- 確度別集計
- 取引登録・紐付け操作を実行していないことの明記

## 品質チェック

1. テーブル行数 = 対象証憑件数。
2. `receipt_id` 重複なし。
3. `取引内容` に `ID=` や `税区分コード=` を残さない。
4. `推測される新規取引(ドラフト)` と `取引内容` のキー構成が全行で一致。
5. ラベル未解決件数を補足に記載。
6. 取引登録・紐付け操作を実行していないことを明記。
7. `取引登録済み` が `[x]` の件数と補足「登録済み取引の識別」の件数が一致。
8. `証憑紐付け済み` は全行 `[ ]`（本スキルでは紐付けを実行しないため）。
